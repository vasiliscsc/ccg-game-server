Platform API Design

TypeScript / Node.js — three internal modules, one deployable service

Internal Module Structure
Auth Module
spec 2
POST /auth/register
POST /auth/login
POST /auth/refresh
POST /auth/logout
Issues short-lived JWT access tokens
Refresh tokens stored in DB
Passwords hashed with bcrypt
Collection Module
spec 3
GET /cards
GET /collection
GET /decks
POST /decks
PUT /decks/:id
DEL /decks/:id
Card catalog is server-defined
Players own card copies
Deck validation on save
Matchmaking Module
spec 4
POST /queue/join
POST /queue/leave
GET /queue/status
─────────────────
internal only:
POST /session/result
MMR-range based matching
Expands range over wait time
Updates MMR on game end
Shared across all modules:   JWT middleware (auth guard)  ·  Request validation  ·  Error handler  ·  Logger  ·  DB connection pool (PostgreSQL)  ·  Redis client

MMR System

Algorithm: Elo (starting point)
Initial MMR: 1000
K-factor: 32 (adjustable per rank band)
Queue matching:
±50 MMR at queue start
Range expands +25 every 30s
Caps at ±300 before any opponent accepted

Deck Validation Rules

Exactly 30 cards per deck
Max 2 copies of any non-legendary
Max 1 copy of any legendary
All cards must be owned by the player
Deck is validated on save and again at queue join
(re-validation at queue join catches cards removed from collection since last save)

Session Handoff

On match found, Platform API calls Game Server:
POST /internal/session/create
{ p1: { id, mmr, deckList },
  p2: { id, mmr, deckList } }
Platform API stores { sessionId, ws_url } in Redis
Pushes connection info to both clients via queue/status polling or SSE

Death Resolution & Action Ordering

Where deterministic ordering is enforced in the engine

Death Wave Algorithm

ACTION COMPLETES
Handler mutates state, returns events
↓
PUBLISH EVENTS
IEventBus fires subscribers in order
① current player L→R  ② opponent L→R
↓
DEATH CHECK
Collect all minions with currentHealth ≤ 0
Sort: current[0..n] → opponent[0..n] → neutral[0..n]
↓ deaths found
none → next action
PHASE 1 — REMOVE & GRIEVE
For each dead minion (in sort order):
remove from board, add GraveyardEntry,
publish MinionDiedEvent
↓
PHASE 2 — DEATHRATTLES
For each dead minion (same sort order):
enqueue deathrattle actions → process fully
New deaths here → queued for next wave
↓
PHASE 3 — REBORNS
For each dead minion with Reborn (same sort order):
enqueue SummonMinionAction at 1 HP
Always after ALL deathrattles of this wave
↓
LOOP — next wave
Run Death Check again
Repeat until no deaths remain
Then: win condition check

Where Each Ordering Is Enforced

IEventBus — trigger fire order
Maintains two subscriber lists per event type: current player (sorted by board index) and opponent (sorted by board index). Always fires current player's list first. Minions register on summon — position order is preserved naturally.
DeathResolutionService — death wave order
Owns the 3-phase wave algorithm. Sorts pending deaths by (player = current/opponent, boardIndex). Enforces: Deathrattles complete before any Reborn summons. New deaths during Phase 2 are deferred to next wave — not processed mid-wave.
IActionQueue — processing order
FIFO queue. Enqueue for normal effects. EnqueueFront reserved for death resolution inserting immediate system actions. Ensures effects enqueued by deathrattles are processed before the next player action.
GameEngine main loop — isolation
Processes one action at a time. Does not start the next action until: current action's handler completes → events published → death check runs to stability. No parallel processing.
ICardHandler.OnDeath — per-minion ordering
Called by DeathResolutionService in sort order. Responsible for enqueuing this minion's deathrattle actions. Does not enqueue Reborn — that is always handled by DeathResolutionService in Phase 3.
Key invariant
A minion's Reborn summon can never happen before its own Deathrattle resolves, and neither can happen before the minion is fully removed from the board. This is enforced by DeathResolutionService phasing, not by individual ICardHandler implementations.

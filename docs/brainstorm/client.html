Client Design

Unity or Godot — a pure event renderer with two server connections

Screen State Machine
Login
register / sign in
→
Main Menu
play / collection / settings
→
Collection
browse cards, build decks
→
Queue
select deck, searching...
→
Game Board
live match
→
Post-Game
result, MMR delta
Post-Game → Main Menu  |  any screen → Login (on token expiry or logout)
Connection Architecture
REST Connection (Platform API)
Active during: Login → Queue
Stateless HTTP with JWT header
Auth, collection, deck CRUD
Queue join / leave / status polling
Polling interval: every 2–3s while in queue
to check for match found notification
WebSocket Connection (Game Server)
Active during: Game Board only
Authenticated with player token from Platform API
Sends: action messages only
Receives: game events stream
On disconnect: auto-reconnect with same token
within the 60s grace window
Event Handling Pattern
// Client maintains a local GameState built entirely from server events
onEvent(event) {
switch (event.type) {
CardDrawn → update hand display, animate draw
CardPlayed → remove from hand, trigger card animation
MinionSummoned→ place minion on board
DamageDealt → show damage number, update health bar
MinionDied → play death animation, remove from board
TurnStarted → highlight active player, start timer
GameEnded → show result overlay, transition to Post-Game
}
}
// Client NEVER mutates game state speculatively — always waits for server event
// This keeps client logic simple: it is a view over an event stream, nothing more

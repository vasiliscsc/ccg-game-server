Backend Architecture Approaches

Three ways to structure the server-side systems — pick the one that fits your team and timeline

A

Modular Monolith (C# / .NET)

One ASP.NET Core application handles auth, collections, matchmaking, and game sessions. Clean internal module boundaries, but deployed as a single binary.

Pros

  • Simplest to develop and deploy
  • No inter-service networking overhead
  • Easy to debug and trace
  • Modules can be extracted later

Cons

  • Scaling is all-or-nothing
  • Game sessions and REST API compete for same process
  • Harder to use TypeScript for faster API iteration
B

Core Split — Game Server (C#) + Platform API (TypeScript) ⭐ Recommended

Two services with a clean seam: a C# game server manages WebSocket connections and live game state; a TypeScript API handles auth, collections, matchmaking, and REST traffic.

Pros

  • Natural seam — game sessions vs. HTTP requests are genuinely different
  • C# game server shares language with Unity (if chosen)
  • TypeScript API is fast to iterate on
  • Each service scales independently
  • Not complex enough to cause deployment pain

Cons

  • Two repos/deployments to manage
  • Requires inter-service auth (JWT shared secret or service tokens)
C

Full Microservices (Auth / Collection / Matchmaking / Game)

Each subsystem is its own independently deployable service with its own database.

Pros

  • Maximum scaling flexibility
  • Clear ownership per team/service

Cons

  • Premature for 1k–10k concurrent scale
  • Significant operational overhead (service mesh, distributed tracing)
  • Much slower to build and debug
